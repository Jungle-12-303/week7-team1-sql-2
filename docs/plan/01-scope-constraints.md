# 01-scope-constraints

## 필수 범위(In Scope)
- `.data` 저장 포맷을 커스텀 텍스트에서 바이너리로 전환
- `INSERT` 시 ID 자동 부여
- 부여된 ID를 B+ 트리 인덱스에 즉시 등록
- `WHERE id = ?` 질의는 인덱스 경로로 조회
- 1,000,000건 이상 삽입 후 성능 비교(인덱스 조회 vs 선형 탐색)
- 기존 텍스트 데이터가 있다면 일회성 마이그레이션 경로 제공
- 단위/기능 테스트 추가 및 회귀 확인
- README에 구현/검증/데모 절차 반영

## 비범위(Out of Scope)
- 디스크 기반 영속 B+ 트리 구현
- 다중 컬럼/복합 인덱스 지원
- SQL 문법 전면 확장(새로운 대규모 파서 기능 추가)
- 분산 처리, 동시성 제어 고도화
- 텍스트/바이너리 동시 쓰기(dual-write) 운영

## 제약 조건
- 구현 언어: C
- 기존 SQL 실행 흐름을 최대한 유지하고 저장 계층만 교체
- `WHERE id = ?` 최적화는 명시적 분기 방식으로 구현
- 테스트와 검증 결과는 발표 가능한 형태로 남긴다.
- 코드 작성 시 한국어 주석을 기본으로 하고, 핵심 분기/자료구조 처리에는 의도와 동작을 이해할 수 있게 친절히 설명한다.

## 수용 기준(완료 조건)
- 수용 기준 1: 바이너리 저장 후 재조회 시 레코드 값이 손실 없이 복원된다.
- 수용 기준 2: `INSERT` 후 같은 레코드를 ID 조회 시 즉시 검색된다.
- 수용 기준 3: `WHERE id = ?` 실행 경로가 선형 탐색과 분리돼 있다.
- 수용 기준 4: 1,000,000건 데이터에서 인덱스 조회가 선형 탐색보다 빠르다는 측정 결과가 기록된다.
- 수용 기준 5: 주요 함수 단위 테스트와 시나리오 기능 테스트가 통과한다.
- 수용 기준 6: README만으로 데모 절차 설명이 가능하다.
