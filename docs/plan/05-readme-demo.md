- 섹션 3: 시스템 흐름(`INSERT -> binary write -> ID -> index`, `WHERE id -> index_find`)

- 섹션 7: 실행 방법(빌드/테스트/벤치/마이그레이션)

## 4분 데모 순서
- 0:00~0:40: 문제 정의와 목표
- 0:40~1:20: 바이너리 저장 전환 포인트 설명
- 1:20~2:20: 동작 데모(INSERT + id 조회 + 일반 조회)
- 2:20~3:20: 테스트 근거와 안정성
- 3:20~4:00: 성능 비교 결과와 결론

## 1분 Q&A 대비 포인트
- 질문 1: 왜 id만 인덱스 최적화했나?
  - 답변: 과제 핵심 요구와 가장 직접 연결되며 검증 가능성이 높기 때문.
- 질문 2: 인덱스 실패 시 어떻게 처리하나?
  - 답변: 에러를 반환하고 필요 시 선형 탐색 fallback 정책으로 안정성 유지.
- 질문 3: 100만 건에서 병목은 무엇인가?
  - 답변: 데이터 생성/IO/탐색 경로별 병목을 분리 측정해 설명.
- 질문 4: 왜 텍스트 대신 바이너리를 선택했나?
  - 답변: 100만 건 삽입/조회에서 파싱 비용을 줄이고, 인덱스 효과를 더 명확히 검증하기 위해서.
