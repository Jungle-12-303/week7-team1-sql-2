# bptree

## 핵심 질문

이번 구현에서 많이 헷갈리는 부분은 아래 두 가지입니다.

1. 차수는 어떻게 결정하는가?
2. `BPTREE_MAX_KEYS`가 커질수록 정말 트리 깊이가 작아지는가?

결론부터 말하면, **네, 일반적으로 `BPTREE_MAX_KEYS`가 커질수록 B+ 트리의 깊이는 얕아집니다.**

---

## 1. 차수란 무엇인가

B+ 트리에서 차수는 쉽게 말해 **노드 하나가 얼마나 많은 자식을 가질 수 있는지**와 관련된 개념입니다.

그런데 구현이나 책마다 차수를 조금 다르게 정의합니다.

- 어떤 자료는 차수를 `최대 자식 수`로 설명합니다.
- 어떤 자료는 차수를 `최대 key 수`와 연결해서 설명합니다.

그래서 발표나 README에서는 헷갈리지 않게 **우리 코드 기준으로 직접 설명하는 것**이 가장 좋습니다.

예를 들면:

> 우리 구현에서는 노드당 최대 key 수를 `BPTREE_MAX_KEYS`로 정의했고, 내부 노드는 최대 `BPTREE_MAX_KEYS + 1`개의 자식을 가질 수 있습니다.

---

## 2. 우리 코드에서 `BPTREE_MAX_KEYS = 31`의 의미

우리 코드에서 `BPTREE_MAX_KEYS = 31`이라면:

- 한 노드에 key를 최대 31개 저장할 수 있고
- 내부 노드는 자식을 최대 32개까지 가질 수 있습니다

즉, 한 번 아래로 내려갈 때마다 최대 32개의 갈래 중 하나를 선택할 수 있다는 뜻입니다.

쉽게 비유하면:

- `MAX_KEYS = 3`이면 작은 서랍장
- `MAX_KEYS = 31`이면 큰 서랍장

큰 서랍장은 한 칸에 더 많이 담을 수 있으니, 같은 양의 데이터를 저장할 때 서랍장을 덜 쌓아도 됩니다.

---

## 3. `BPTREE_MAX_KEYS`가 커지면 왜 깊이가 작아지는가

트리의 깊이는 데이터가 많아질수록 커집니다.
하지만 한 노드가 많은 key와 많은 자식을 담을 수 있으면, 트리가 한 층에서 커버할 수 있는 범위가 훨씬 넓어집니다.

예를 들어:

- 작은 노드는 적은 데이터만 나눠서 관리할 수 있음
- 큰 노드는 많은 데이터를 넓게 나눠서 관리할 수 있음

그래서:

- `BPTREE_MAX_KEYS`가 작으면 트리가 빨리 아래로 길어짐
- `BPTREE_MAX_KEYS`가 크면 트리가 넓게 퍼지면서 깊이는 덜 늘어남

즉, **한 노드가 클수록 전체 트리는 더 얕아질 가능성이 큽니다.**

---

## 4. 그림으로 이해하기

### `MAX_KEYS`가 작은 경우

노드 하나가 많이 못 담으니까 층이 더 쉽게 늘어납니다.

```text
        [50]
      /      \
    [20]     [80]
   /   \     /   \
 ...
```

### `MAX_KEYS`가 큰 경우

노드 하나가 넓은 범위를 커버할 수 있으니 깊이가 덜 필요합니다.

```text
[10 20 30 40 50 60 70 80 90 ...]
 /  /  /  /  /  /  /  /  /
...
```

비유하면:

- 작은 차수: 나뭇가지가 적은 대신 층이 많아짐
- 큰 차수: 나뭇가지가 많은 대신 층이 적어짐

---

## 5. 큰 값으로 잡으면 좋은 점

- 트리 깊이가 얕아짐
- 탐색할 때 내려가는 단계 수가 줄어듦
- 대용량 데이터에서 더 유리할 수 있음

즉, `WHERE id = ?` 같은 인덱스 탐색에서 비교 횟수를 줄이는 데 도움이 됩니다.

---

## 6. 너무 크게 잡으면 생기는 단점

그렇다고 무조건 크게 잡는 것이 항상 좋은 것은 아닙니다.

- 노드 하나의 크기가 커짐
- 삽입 시 내부 배열에서 key를 밀어 넣는 비용이 커질 수 있음
- split 로직 이후 한 번에 다뤄야 할 메모리 양이 많아짐
- 구현과 디버깅이 조금 더 부담될 수 있음

즉, 차수는 단순히 크게만 정하는 것이 아니라 **깊이를 줄이는 장점**과 **노드 하나를 다루는 비용** 사이에서 균형을 잡아 결정합니다.

---

## 7. 이번 과제 관점에서 이해하면

이번 과제는 `WHERE id = ?`일 때 B+ 트리 인덱스를 사용하는 것이 핵심입니다.

이때 `BPTREE_MAX_KEYS`가 충분히 크면:

- 트리 높이가 낮아지고
- 루트에서 리프까지 내려가는 횟수가 줄어들고
- 대규모 데이터에서도 `id` 검색이 빨라질 수 있습니다

반대로 `name` 같은 인덱스가 없는 컬럼은 B+ 트리를 타지 못하고 선형 탐색을 하게 됩니다.

그래서 발표할 때는 아래처럼 비교해서 설명하면 좋습니다.

- `WHERE id = ?`
  B+ 트리 인덱스 사용
- `WHERE name = ?`
  인덱스가 없으면 전체 레코드 선형 탐색

---

## 8. 발표 때 바로 써도 되는 정리 문장

아래 문장을 그대로 써도 자연스럽습니다.

> 우리 구현에서는 노드당 최대 key 수를 `BPTREE_MAX_KEYS`로 정의했습니다. 이 값이 커질수록 내부 노드가 더 많은 자식을 가질 수 있어 한 단계에서 커버하는 범위가 넓어집니다. 따라서 전체 데이터가 같을 때 B+ 트리의 깊이는 일반적으로 더 얕아집니다. 다만 노드 하나의 크기가 커지므로 삽입 시 배열 정렬과 메모리 처리 비용은 조금 증가할 수 있습니다.

---

## 한 줄 요약

`BPTREE_MAX_KEYS`가 커질수록 한 노드가 더 많은 key와 자식을 담을 수 있어서, B+ 트리는 보통 더 얕아집니다. 하지만 노드 하나를 다루는 비용은 조금 커질 수 있으므로 적절한 균형이 필요합니다.
