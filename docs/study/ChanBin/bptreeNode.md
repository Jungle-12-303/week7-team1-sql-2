# bptreeNode

## 한 줄 설명

`BptNode`는 **B+ 트리의 노드 한 칸**을 나타내는 구조체입니다.
쉽게 말하면 트리 안에 들어 있는 **박스 하나**라고 생각하면 됩니다.

---

## 구조체 코드

```c
struct BptNode {
    bool is_leaf;
    size_t size;
    uint64_t keys[BPTREE_MAX_KEYS];
    union {
        struct {
            RowRef refs[BPTREE_MAX_KEYS];
            BptNode *next;
        } leaf;
        struct {
            BptNode *children[BPTREE_MAX_KEYS + 1];
        } internal;
    } as;
};
```

---

## 필드별 설명

### 1. `is_leaf`

```c
bool is_leaf;
```

이 노드가:

- 리프 노드인지
- 내부 노드인지

구분하는 값입니다.

- `true`면 맨 아래 실제 데이터를 가리키는 리프 노드
- `false`면 위쪽에서 길을 안내하는 내부 노드

예:

```text
[30 | 50]   <- 내부 노드
[10 | 20]   <- 리프 노드
```

---

### 2. `size`

```c
size_t size;
```

현재 이 노드 안에 **실제로 몇 개의 key가 들어 있는지**를 의미합니다.

예:

```text
[10 | 20 | 30]
```

이면 `size = 3`입니다.

즉 배열 전체 크기가 아니라, **현재 사용 중인 개수**입니다.

---

### 3. `keys`

```c
uint64_t keys[BPTREE_MAX_KEYS];
```

이 배열에는 노드가 관리하는 **정렬된 key 값들**이 저장됩니다.
우리 코드에서는 이 key가 곧 `id`입니다.

예:

```text
keys = [10, 20, 30]
```

뜻:

- 이 노드는 `10`, `20`, `30`이라는 id를 관리하고 있음

항상 정렬된 상태여야 검색과 삽입이 쉬워집니다.

---

## `union as`의 의미

```c
union {
    ...
} as;
```

이 부분은 이 노드가 **리프 노드일 때의 모습**과 **내부 노드일 때의 모습** 중 하나만 사용한다는 뜻입니다.

- 리프 노드면 `as.leaf`를 사용
- 내부 노드면 `as.internal`을 사용

이렇게 하면 메모리를 아끼고 구조를 하나로 통일할 수 있습니다.

---

## 리프 노드일 때

```c
struct {
    RowRef refs[BPTREE_MAX_KEYS];
    BptNode *next;
} leaf;
```

### `refs`

```c
RowRef refs[BPTREE_MAX_KEYS];
```

각 key가 가리키는 **실제 데이터 위치**를 저장합니다.

예:

```text
keys = [10, 20, 30]
refs = [A,  B,  C]
```

뜻:

- `id 10`의 실제 row 위치는 `A`
- `id 20`의 실제 row 위치는 `B`
- `id 30`의 실제 row 위치는 `C`

즉 리프 노드는 `id -> 실제 데이터 위치`를 가지고 있습니다.

### `next`

```c
BptNode *next;
```

다음 리프 노드를 가리키는 포인터입니다.

예:

```text
[10 20] -> [30 40] -> [50 60]
```

이 연결 덕분에:

- 범위 검색이 쉬워지고
- 순서대로 데이터를 읽기 편해집니다

예를 들어 `30 이상`을 찾고 싶다면 `[30 40]` 리프에서 시작해서 `next`를 따라가면 됩니다.

---

## 내부 노드일 때

```c
struct {
    BptNode *children[BPTREE_MAX_KEYS + 1];
} internal;
```

### `children`

```c
BptNode *children[BPTREE_MAX_KEYS + 1];
```

이 배열은 아래 자식 노드들을 가리킵니다.

왜 `+1`이냐면:

- key가 `n`개면
- 자식은 `n + 1`개가 필요하기 때문입니다

예를 들어 내부 노드가:

```text
[30 | 50]
```

이면 자식은 3개가 필요합니다.

```text
          [30 | 50]
         /    |    \
 [10 20] [30 40] [50 60]
```

즉:

- `children[0]` : 30보다 작은 값
- `children[1]` : 30 이상 50 미만
- `children[2]` : 50 이상

---

## 전체 구조를 한 번에 이해하기

### 경우 1. 리프 노드

예:

```text
[10 | 20 | 30]
```

구조:

```text
is_leaf = true
size = 3
keys = [10, 20, 30]
refs = [ref10, ref20, ref30]
next = 다음 리프 노드
```

뜻:

- 이 노드는 맨 아래 노드
- 실제 row 위치를 가지고 있음

### 경우 2. 내부 노드

예:

```text
[30 | 50]
```

구조:

```text
is_leaf = false
size = 2
keys = [30, 50]
children[0] = 왼쪽 자식
children[1] = 가운데 자식
children[2] = 오른쪽 자식
```

뜻:

- 실제 데이터 위치는 들고 있지 않음
- 아래 어느 자식으로 내려갈지만 알려줌

---

## 검색 예시

트리가 아래처럼 있다고 가정합니다.

```text
          [30 | 50]
         /    |    \
 [10 20] [30 40] [50 60]
```

`id = 40`을 찾는 과정:

1. 루트 `[30 | 50]`를 본다
2. `40`은 `30 이상 50 미만`이므로 가운데 자식으로 내려간다
3. 리프 `[30 40]`에서 `40`을 찾는다
4. 같은 위치의 `refs`에서 실제 row 위치를 얻는다

즉:

- 내부 노드의 `keys`는 길 안내
- 리프 노드의 `keys + refs`는 실제 검색 결과

---

## 삽입 예시

처음 리프가 아래처럼 있다고 가정합니다.

```text
[10 20 30]
```

여기에 `40`을 넣는데 노드가 꽉 차 있으면 split이 발생합니다.

```text
[10 20] -> [30 40]
```

그리고 부모 내부 노드가 생깁니다.

```text
       [30]
      /    \
 [10 20] [30 40]
```

이때:

- 리프 노드는 `refs`와 `next`를 가짐
- 내부 노드는 `children`을 가짐

즉 이 구조체 하나로 split 전후의 모든 노드를 표현할 수 있습니다.

---

## 왜 `union`을 쓰는가

리프 노드에는 `refs`가 필요하지만 `children`은 필요 없습니다.
반대로 내부 노드에는 `children`이 필요하지만 `refs`는 필요 없습니다.

그래서 둘 다 항상 들고 있는 대신:

- 리프일 때는 `leaf`만 사용
- 내부일 때는 `internal`만 사용

이렇게 해서 메모리를 아끼고 구조를 깔끔하게 만든 것입니다.

---

## 한 줄 요약

`BptNode`는 B+ 트리의 한 노드를 나타내는 구조체이며, 리프 노드일 때는 `id`와 실제 데이터 위치를 저장하고, 내부 노드일 때는 어느 자식으로 내려갈지 알려주는 역할을 합니다.
