## 2. 평가 기준

과제 가이드의 핵심 요구사항을 바탕으로 아래 항목으로 평가했다.

| 항목 | 배점 | 설명 |
| --- | ---: | --- |
| 과제 적합성 | 15 | 현재 구현이 과제 요구와 얼마나 가까운지 |
| B+ 트리 연결 용이성 | 15 | 자동 ID, 인덱스 등록, `WHERE id = ?` 분기 추가가 쉬운지 |

## 4. 최종 추천

가장 좋은 기반 프로젝트는 **혜연의 `wk06-mini-sql`**이다.

추천 이유는 아래와 같다.

- `INSERT`, `SELECT`, `WHERE column = value`, `[schema.]table` 구조가 이미 갖춰져 있어 과제 요구와 가장 가깝다.
- 저장 계층 구조가 비교적 단순하고 명확해서 `자동 ID 부여 -> B+ 트리 등록 -> WHERE id 인덱스 조회` 흐름을 얹기 쉽다.
- README가 가장 발표 친화적이라 목요일 데모 준비까지 고려했을 때 유리하다.
- 테스트 스크립트와 기능 검증 흐름이 이미 있어 B+ 트리 추가 후 회귀 테스트를 만들기 좋다.

## 5. 후보별 상세 평가

### 5.1 혜연

저장소: https://github.com/iamnuked/wk06-mini-sql

- 과제 적합성: 13/15
  - `INSERT`, `SELECT`, `SELECT *`, `WHERE column = value`를 이미 지원한다.
  - README 예시에도 `WHERE id = 2`가 직접 등장해 과제와 가장 자연스럽게 이어진다.
  - 다만 현재는 자동 ID 부여와 B+ 트리 자체는 아직 없다.
- B+ 트리 연결 용이성: 14/15
  - 저장 흐름이 `append_insert_row`, `run_select_query`처럼 분명해 인덱스 삽입 지점과 조회 분기 지점을 찾기 쉽다.
  - `.schema` / `.data` 구조가 단순해서 `id` 전용 인덱스 파일 또는 메모리 구조를 붙이기 편하다.
  - 현재 구조상 `WHERE id = ?`만 특수 처리하는 확장이 가장 자연스럽다.
- 테스트 기반: 8/10
  - `Makefile`, `tests/test_runner.c`, `scripts/test.ps1`가 있다.
  - 기능 테스트 흐름이 있어 B+ 트리 도입 후 회귀 검증을 추가하기 좋다.
  - 다만 테스트가 세분화된 구조는 아니라 더 촘촘히 늘려야 한다.
- README/발표 준비도: 5/5
  - README에 동작 흐름, 모듈 설명, 데모 절차가 잘 정리돼 있다.
  - 과제 발표를 README 기반으로 해야 한다는 점에서 가장 강하다.
- 대용량 확장성: 3/5
  - 현재는 순차 탐색 구조라 100만 건 벤치마크는 별도 스크립트를 추가해야 한다.
  - 그래도 구조가 단순해 성능 비교 코드를 넣는 것은 어렵지 않다.

### 5.2 정연

저장소: https://github.com/Jungle-12-303/week6-team7-sql-processor

- 과제 적합성: 11/15
  - `INSERT`, `SELECT`, `SELECT ... WHERE`는 구현돼 있다.
  - 다만 `SELECT *`를 지원하지 않고 저장 구조도 비교적 단순한 CSV 중심이다.
  - 과제 연결은 충분히 가능하지만 혜연보다 현재 상태가 요구사항에 덜 맞닿아 있다.
- B+ 트리 연결 용이성: 13/15
  - `cli_runner`, `parser`, `executor`, `storage` 분리가 깔끔하다.
  - 인덱스 계층을 추가하기에는 좋은 구조다.
  - 다만 자동 ID 정책과 별도 메타데이터 설계는 조금 더 작업이 필요하다.
- 테스트 기반: 10/10
  - `unit` / `integration` 테스트가 분리돼 있어 가장 탄탄하다.
  - B+ 트리 도입 후 회귀 테스트를 붙이기에 네 후보 중 가장 좋다.
- README/발표 준비도: 4/5
  - 설명이 잘 돼 있고 문서도 충분하다.
  - 발표용 직관성은 혜연보다 약간 덜하지만 여전히 우수한 편이다.
- 대용량 확장성: 2/5
  - 대량 데이터 생성이나 성능 실험용 준비는 거의 없다.
  - 벤치마크 코드는 새로 만들어야 할 가능성이 크다.

### 5.3 찬빈

저장소: https://github.com/Krafton-Jungle-Week6-sqlparser/SqlParser.git

- 과제 적합성: 10/15
  - `INSERT`, `SELECT *`, 모듈화된 구조는 장점이다.
  - 하지만 README에는 `WHERE` 미지원으로 적혀 있어 문서와 구현 상태가 완전히 일치하지 않는다.
  - 과제 기반으로 사용하려면 현재 지원 범위를 먼저 다시 검증해야 한다.
- B+ 트리 연결 용이성: 12/15
  - `sql / execution / storage / schema` 구조가 잘 나뉘어 있어 확장성은 좋다.
  - 다만 지금 상태에서 바로 과제를 붙이기엔 구현 범위 불확실성이 있다.
- 테스트 기반: 8/10
  - 테스트 러너와 관련 문서는 있다.
  - 검증 문화는 보이지만 정연만큼 세분화되진 않았다.
- README/발표 준비도: 4/5
  - README와 발표용 문서가 따로 있어 발표 준비에는 유리하다.
  - 다만 문서와 실제 지원 범위가 맞지 않을 수 있다는 점이 약점이다.
- 대용량 확장성: 2/5
  - 100만 건 삽입이나 인덱스/비인덱스 성능 비교를 위한 기반은 거의 없다.

## 7. 결론

이번 과제는 단순히 SQL 처리기를 확장하는 것을 넘어서, `자동 ID 부여`, `B+ 트리 인덱스 등록`, `WHERE id = ?` 인덱스 조회, `100만 건 이상 성능 비교`, `테스트 및 README 발표`까지 한 번에 완성해야 한다.

이 기준으로 보면 **혜연 프로젝트가 가장 균형이 좋다.**

- 현재 구현 범위가 과제 요구와 가장 가깝고
- B+ 트리 연결 지점이 명확하며
- README와 데모 흐름이 가장 잘 준비되어 있고
- 팀원들이 빠르게 이해하고 같이 작업하기에도 가장 무난하다.

따라서 팀 기반 구현 시작점으로는 **혜연 저장소를 사용하는 것을 추천한다.**
